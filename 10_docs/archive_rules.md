# 本项目归档规则

本规则用于规范 Ye Wenguang 硕士毕业归档项目的目录、文件、格式、敏感信息和大文件处理方式。

## 一、目录命名规则

1. 一级目录使用数字前缀和英文说明，保持现有结构不随意更名。
2. 方法目录按论文章节命名，例如 `chapter3_cross_model_codegen/`。
3. 数据集和实验目录按数据集或实验类型命名，例如 `HumanEval/`、`RepoExec/`、`EvoCodeBench/`、`ablation/`。
4. 新增目录应先判断是否能放入现有目录，避免重复创建含义相近的目录。

## 二、文件命名规则

1. 推荐使用小写英文、数字、下划线和短横线。
2. 文件名应体现章节、材料类型、版本和日期。
3. 不建议使用 `final2`、`new`、`latest`、`copy` 等无法追踪的名称。
4. 示例：`chapter4_trace_repair_ablation_results_v1_20260513.csv`。

## 三、格式规则

1. 说明文档优先使用 Markdown。
2. 配置文件优先使用 YAML 或 JSON。
3. 表格数据优先使用 CSV、XLSX 或 Markdown 表格。
4. 原始模型输出建议使用 JSONL、JSON 或 TXT。
5. 图表建议同时保存源文件和导出文件。

## 四、压缩包限制

1. 不建议只提交压缩包而不解压说明。
2. 如必须提交压缩包，应在同目录提供说明文件，写明压缩包内容、来源、生成方式和校验值。
3. 大型压缩包应与导师确认是否进入 Git 仓库。
4. 压缩包内不得包含真实密钥、隐私数据或无关缓存。

## 五、敏感信息处理

1. 不得提交 API Key、访问令牌、密码、私有连接串和账号信息。
2. 私有仓库地址、服务器地址和内部接口如需记录，应脱敏处理。
3. 涉及个人信息、评审隐私或未公开材料时，应按学院要求控制访问范围。
4. 脱敏前后如影响复现，应在说明文件中写明处理方式。

## 六、二进制文件处理

1. PDF、PPT、图片、视频和扫描件可以归档，但需命名清楚。
2. 可编辑源文件应与二进制导出文件同时保存。
3. 视频和大型图片应控制体积，必要时使用外部存储并记录链接和校验值。
4. 不应提交系统构建产物、缓存、临时文件和日志垃圾文件。

## 七、大文件处理

1. 大型数据集、模型文件、仓库镜像和视频应先评估是否适合进入 Git。
2. 如使用外部存储，需在对应目录保存说明文件，包含存储位置、访问权限、校验值、负责人和备份状态。
3. 若使用 Git LFS，应在版本日志中记录启用范围和文件类型。
4. 大文件不应影响普通检查者打开和浏览归档目录。

## 八、更新规则

1. 新增材料后运行 `python3 tools/generate_tree.py` 更新 `tree.txt`。
2. 阶段检查前运行 `python3 tools/check_missing_items.py`。
3. 更新 `00_admin/archive_status_table.md` 和 `00_admin/version_log.md`。
4. 如材料替换最终版本，应保留替换原因和日期。
