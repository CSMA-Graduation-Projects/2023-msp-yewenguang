# 第 4 章：基于程序运行轨迹的多智能体代码生成与修复方法

## 章节主题

本目录归档论文第 4 章“基于程序运行轨迹的多智能体代码生成与修复方法”相关材料。该章节利用程序执行过程中的运行轨迹、异常信息、变量状态和测试反馈，组织多个智能体完成错误定位、原因分析、补丁生成、验证和回滚。

## 核心智能体

### 插桩智能体

插桩智能体负责在目标代码中加入必要的跟踪逻辑，采集函数调用、变量值、分支路径、异常栈、输入输出和测试执行结果。归档时需保留插桩规则、插桩前后代码样例、轨迹格式和开销统计。

### 分析智能体

分析智能体负责读取运行轨迹和测试失败信息，判断失败原因、定位疑似错误语句、推断错误传播路径，并生成面向修复智能体的结构化分析结果。归档时需保留分析提示词、分析输出、错误分类和案例解释。

### 修复智能体

修复智能体根据原始任务、当前代码、运行轨迹、分析结论和历史经验生成补丁。归档时需保留每轮补丁、补丁应用记录、测试结果和最终采纳依据。

## 历史经验学习

本章方法需要整理历史失败和成功修复案例，形成可复用经验。归档内容应包括：

- 错误类型与修复策略映射。
- 历史案例索引和检索方式。
- 经验被调用的轮次、触发条件和对修复结果的影响。
- 历史经验加入前后的消融对比。

## 回滚机制

当补丁导致测试结果退化、引入新异常或破坏原有功能时，应触发回滚机制。归档时需保留：

- 回滚判定规则。
- 补丁版本记录。
- 回滚前后测试对比。
- 多轮修复中的状态转移日志。

## 关键输入输出

| 类型 | 内容 | 说明 |
|---|---|---|
| 输入 | 初始代码、任务描述、测试用例 | 应保留原始数据来源 |
| 输入 | 插桩配置和运行环境 | 放入 `configs/` |
| 中间产物 | 运行轨迹、异常栈、变量状态 | 放入 `data/` 或 `results/trace_logs/` |
| 中间产物 | 分析智能体输出、修复建议、补丁版本 | 应保持轮次可追踪 |
| 输出 | 最终修复代码、测试通过记录、失败案例 | 对应论文实验结果 |

## 实验与结果要求

需要归档以下实验材料：

1. 轨迹采集实验：插桩覆盖信息、轨迹大小、采集开销。
2. 修复性能实验：修复成功率、平均轮次、测试通过率、失败类型。
3. 智能体消融实验：去除插桩、去除分析、去除历史经验、去除回滚机制后的效果。
4. 案例分析：至少保留若干个成功修复和失败修复案例，说明运行轨迹如何帮助定位和修复。
5. 成本统计：模型调用次数、token 使用、运行时间和测试执行时间。
